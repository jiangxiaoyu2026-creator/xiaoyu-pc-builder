# 智能装机 (AI Builder) 小程序开发经验与避坑指南

> **文档创建时间**: 2026-04-26
> **作用**: 记录小程序“智能装机”模块在开发过程中的核心架构、UI 规范以及踩过的坑。下次开启新窗口或新 Agent 时，请**首先阅读本文件**以无缝继承开发上下文。

---

## 1. 核心设计规范 (UI/UX)
*   **微信暗黑模式 (WeChat Dark Mode)**：抛弃了臃肿的卡片（Bento UI）设计。全面采用**极简紧凑列表 (Cell List)**。
*   **背景与边框**：深色模式底色使用 `#191919` 或 `#111111`，列表项之间只用 `0.5px` 的半透明白线 (`rgba(255,255,255,0.05)`) 分割。
*   **关键颜色**：
    *   **价格高亮**：统一使用微信原生红 `#FA5151`。
    *   **推荐角标**：橙色 `#ff9500`。
    *   **等宽数字**：凡是涉及到价格、算力、跑分的数字，必须使用 `font-family: ui-monospace, monospace;` 营造极客/终端感。

## 2. Taro 小程序开发“天坑”记录

### 2.1 ScrollView 布局失效问题 (极度重要)
*   **Bug 描述**：在 BottomSheet（底部抽屉）中，使用 `ScrollView` 作为纵向滚动列表时，如果外层是 `flex` 容器，`ScrollView` 会无限撑大，导致无法滚动，且会把底部的按钮挤出屏幕外。
*   **解决方案**：`ScrollView` 的类名（如 `.sheet-body`）必须同时设置：
    ```scss
    .sheet-body {
      flex: 1;
      height: 0; /* 必须加！否则 Taro 无法计算正确的滚动视图高度 */
      min-height: 0;
    }
    ```

### 2.2 横向滚动 (ScrollX) 文字换行与挤压
*   **Bug 描述**：商品品牌列表如果太长，会自动折行，导致布局崩坏。
*   **解决方案**：给外层父容器设置 `white-space: nowrap;`，给内层的子元素设置 `flex-shrink: 0;`。

### 2.3 隐藏底部 TabBar 导致的“死胡同”
*   **经验**：在“智能装机”这种核心 Tab 页面使用 `Taro.hideTabBar()` 可以获得极佳的全屏沉浸感（特别是底部有自定义的 DeFi 计价控制台时）。
*   **坑点**：一旦隐藏，用户将**无法切换 Tab**。
*   **规范**：隐藏 TabBar 的页面，**必须**在底部的悬浮控制台或页面显眼处，提供一个原生的返回链接（如 `←首页`），调用 `Taro.switchTab({ url: '/pages/index/index' })` 放用户出去。

### 2.4 Taro 组件按需引入 (致命白屏 Bug)
*   **Bug 描述**：如果在 JSX 中使用了某个微信原生/Taro 基础组件（如 `<Canvas>`、`<Picker>` 等），但忘记在文件顶部 `import { Canvas } from '@tarojs/components'`，Taro/Webpack **编译时往往不会报错**，但一进入该页面就会引发 React 运行时的 `ReferenceError: Canvas is not defined`，导致**整个页面瞬间白屏崩溃**。
*   **教训**：每次添加新的 UI 组件前，务必检查顶部是否已经将其从 `@tarojs/components` 引入。

## 3. 业务逻辑与数据结构
*   **缺货逻辑 (Out of Stock)**：
    *   API 返回的硬件列表中，如果 `price === 0`，代表缺货。
    *   **前端处理**：DOM 必须追加 `.out-of-stock` 类（`opacity: 0.4; filter: grayscale(100%); pointer-events: none;`），并将价格文本硬编码替换为“缺货”。
*   **商品排序**：
    *   打开分类抽屉时，带有 `isRecommended: true` 的商品必须**优先置顶**。
    *   默认降级排序为按价格升序排列。

## 4. 后续接手开发建议
1.  **关于 API 数据**：目前的接口对接在 `src/services/api.ts` 中，所有硬件获取走的是 `getProducts({ category, page, page_size })`。如果需要引入新的字段（如功耗详细数据），请先检查后端 Swagger 或使用 `curl` 进行格式确认。
2.  **长列表性能优化**：目前 `buildList` 采用全量渲染，如果未来单个分类下的商品数量超过 500 个，建议将 `ScrollView` 替换为 Taro 的**虚拟列表 (VirtualList)** 组件以防止爆内存。

## 2026-04-26 用户偏好记录
- **按钮设计偏好**: 用户更喜欢偏方正的按钮（如 `border-radius: 4px` 或 `8px`），不喜欢过于圆润的药丸形按钮（如 `border-radius: 40rpx` 或 `20px`）。在后续的 UI 开发中，应默认使用小圆角或直角设计。

## 2026-04-26 架构总结：PC与小程序多端互通策略

项目已实现真正的 **“一云多端” (One Backend, Multiple Frontends)** 架构：

1. **“同宗同源”的后端数据（100% 数据互通）**：
   - PC 端网站、移动端网页版与微信小程序共享同一个 Python FastAPI 后端服务器与数据库。
   - 商品库、游戏帧数数据、价格变动、用户配置单实现全平台多端实时同步。

2. **“血脉相连”的底层代码逻辑（React 生态）**：
   - 前端采用 React (Vite)，小程序采用 Taro 4 (React)。
   - 核心业务逻辑（算力评估、价格汇总、数据请求）高度重合复用，语法结构一致，大幅降低维护成本。

3. **“因地制宜”的 UI 表现（端侧分离）**：
   - **后端与逻辑代码复用，UI 界面按端定制**。
   - PC 端：采用宽屏 Bento 卡片网格与极客悬浮动效。
   - 小程序端：针对移动端操作习惯，重构为 BottomSheet (底部弹窗)、横向滑动过滤芯片、高密度垂直列表。
   - 架构的延展性极强，当前 Taro 代码若有需求，亦可直接编译打包为原生 App (iOS/Android)。
